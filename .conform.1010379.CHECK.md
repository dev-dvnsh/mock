## Task 2 — Project Setup and Argument Parsing

### What to Build (Clearer)

**Project structure:**

- One folder called `mock`
- Inside it: `main.go`, `data.go`, `go.mod`
- That's it. No subfolders needed for this size project.

**`go mod init`:**

- Run this once in your project folder before writing any code
- Name your module something like `github.com/yourusername/mock`
- This creates `go.mod` — don't edit it manually

**What `main.go` needs to do at this stage:**

- Read `os.Args` — this is just a slice of strings, index 0 is the program name, index 1 is your schema string, index 2 is the count
- Check that at least 2 arguments were passed — if not, print how to use the tool and call `os.Exit(1)`
- Take the schema string like `"name, email, age"` and split it by comma into a slice of strings — then trim the spaces off each one so `" email"` becomes `"email"`
- Take the count string like `"50"` and convert it to an integer — handle the case where it's not a valid number
- Look through the remaining arguments for `--format` and `--output` flags — if `--format csv` is there, store `"csv"`, otherwise default to `"json"`. If `--output somefile.json` is there, store the filename.
- At the end of this task, just print everything you parsed so you can verify it works — schema fields as a slice, count as an integer, format, output filename

**The usage message when arguments are wrong should look like:**

```
Usage: mock "<fields>" <count> [--format json|csv] [--output filename]
Example: mock "name, email, age" 100 --format csv
```

## Task 3 — The Generator: Fake Data per Field Type

### What to Build (Clearer)

**Two files involved:**

- `data.go` — holds all your hardcoded lists (names, cities etc). Nothing else.
- `main.go` — holds the generator function that uses those lists

**What goes in `data.go`:**

- A slice of first name strings — at least 20 names
- A slice of last name strings — at least 20 names
- A struct with two fields: City string and Country string
- A slice of that struct — at least 30 city+country pairs where each city is correctly matched to its country

**The generator function in `main.go`:**

- Takes one argument: a field name string like `"name"` or `"email"`
- Returns one string: the generated value
- Inside it, use a switch statement on the field name
- Each case calls a small helper that generates that specific type

**What each generator does specifically:**

- `name` — pick a random index into firstNames slice, pick a random index into lastNames slice, join them with a space
- `email` — generate a name first using your name generator, take the first name part, lowercase it, add a dot, take the first letter of the last name, lowercase it, add 5 random digits, add `@gmail.com`
- `age` — random number between 18 and 65
- `phone` — build a string that looks like `+91-` followed by 10 random digits
- `city` — pick a random index into your cities slice, return just the City field
- `country` — pick the SAME random index you used for city — this is the key part, city and country must be linked, so generate a random index once and use it for both
- `uuid` — build a string in the format `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` where each x is a random hex character (0-9, a-f). Use `fmt.Sprintf` with `%x` format.
- `date` — pick a random year between 1980 and 2023, random month 1-12, random day 1-28 (use 28 max to avoid invalid dates), format as `YYYY-MM-DD`
- `bool` — random int that's either 0 or 1, return `"true"` or `"false"` as a string
- `int` — random number between 1000 and 9999
- default case — return `"unknown"` and print a warning to stderr saying which field name wasn't recognized

**The city+country problem specifically:**

- Your generator function takes only one field name at a time
- But `city` and `country` need to return values from the same entry
- Solution: generate a random index when `city` is called, store it somewhere accessible, use that same index when `country` is called
- Think about where to store that index so both cases can access it — a variable outside the switch? A package-level variable? Think about this one.

## Task 4 — Build the Output: JSON and CSV

### What to Build (Clearer)

**The main loop:**

- You have a slice of field names from Task 2 — `["name", "email", "age"]`
- You have a count — `50`
- Loop count times
- Each iteration: create a map where the keys are field names and the values are generated by calling your generator function
- After the loop you have a slice of maps — one map per record

**For JSON output:**

- Take your slice of maps
- Convert it to a nicely indented JSON string using the json package
- If `--output` was provided, create that file and write the JSON to it
- If no `--output`, just print it to the terminal
- The output should be a JSON array of objects — each object has the field names as keys

**For CSV output:**

- Create a csv writer that writes to either a file or stdout — same logic as JSON
- First write one row that contains all your field names as the header
- Then for each record in your slice, write one row with the values in the same order as the header
- Call Flush at the end or your last rows might not appear
- Order matters for CSV — the header and data rows must have values in the same column order

**The file vs stdout decision:**

- At the start of your output section, decide: is `--output` empty or not?
- If empty, your writer target is `os.Stdout`
- If not empty, create the file with `os.Create` and use that as your writer target
- The rest of the code doesn't change — same write calls either way
- Don't forget to close the file if you opened one

**One thing that will bite you:**

- Go maps don't guarantee key order
- When you write CSV rows, you need the values in the same order as the header every time
- Solution: use your original field names slice as the order — iterate over it to build each CSV row instead of iterating over the map directly

## Task 5 — Install Globally and Polish

### What to Build (Clearer)

**The `--help` flag:**

- At the very start of main, before anything else, check if any argument equals `--help` or `-h`
- If it does, print your usage message and exit with code 0 — not 1, because help is not an error
- The help message should show: description, usage line, all flags with defaults, all supported field types, at least two examples

**Edge cases to handle — go through each one:**

- Count is 0 — print "count must be greater than 0" and exit
- Count is negative — same message
- Count is not a number like `"abc"` — print "count must be a valid integer" and exit
- Schema string is empty `""` — print "schema cannot be empty" and exit
- `--format` value is not `json` or `csv` — print "format must be json or csv" and exit
- Unknown field names — already handled in Task 3 with the default case, but make sure the warning is printed and the field still gets a value so the record isn't broken

**`go install`:**

- Run `go install` from your project folder
- It puts the binary in `$GOPATH/bin` or `$HOME/go/bin`
- Add that folder to your PATH in your `.bashrc` or `.zshrc` — one line: `export PATH=$PATH:$HOME/go/bin`
- Source your shell config or open a new terminal
- Now type `mock` from anywhere — it should work

**`go vet`:**

- Run `go vet ./...` from your project folder
- It catches common mistakes the compiler misses
- Fix everything it reports before calling the project done
